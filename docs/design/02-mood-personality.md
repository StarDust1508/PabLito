# Дизайн-док: подсистема НАСТРОЕНИЯ и ЛИЧНОСТИ Паблито

Статус: предложение к реализации. Код не менялся — это план поверх текущего
`src/core/mood.ts`, `src/core/personality.ts`, `src/hooks/usePablito.ts`.

Владелец сформулировал требование прямо: настроение — не декорация и не
единственная механика ради «вау-эффекта» в демо, а главная фича приложения.
Значит критерий успеха один: **владелец должен на третьей неделе использования
чувствовать, что Паблито помнит его и по-настоящему реагирует**, а не то, что
цифра `energy` анимированно бегает по слайдеру.

Ниже — разбор по пяти пунктам задания.

---

## 0. Что уже есть (баланс, не критика)

Текущая реализация — правильный скелет: состояние в БД, чистые функции
`applyEvents`/`decayToward`, зажатие 0..100, подмешивание в системный промпт
текстом, а не числом («estás con muchísima energía», а не `energy: 92`) —
это важно, потому что модель лучше играет роль по словесному описанию, чем по
сырым цифрам. Слабые места, которые решает этот документ:

1. Три оси одного порядка — всё «настроение момента»; нет ничего, что живёт
   неделями и не сбрасывается дрейфом.
2. События определяются регексами по последней реплике — грубо, без контекста
   разговора и без сигнала от самой модели.
3. Нет истории значимых моментов — Паблито не может сослаться на «то, как ты
   грустил на прошлой неделе из-за работы».
4. Нет явных правил, как настроение меняет *форму* речи (длину, инициативу,
   темы), а не только тон одним предложением в промпте.
5. Нет защиты от залипания в крайностях и нет явной этической рамки — при
   «главная фича = эмоциональная связь» это первое, что должно быть прописано,
   а не додумано потом.

## 1. Модель состояния

### 1.1 Зачем расширять три оси

`energy/warmth/patience` — это состояние *сессии/момента*. У них уже есть
дрейф к `BASELINE` и это правильно: сиюминутное должно забываться. Но живой
друг отличается от настроения тем, что помнит *отношения* — это должно быть
отдельной, гораздо более инертной величиной, которая не «дрейфует к нейтрали»
между сессиями, а копится.

Смешивать это в одни три оси — ошибка: если `warmth` одновременно и
«сейчас я тёплый, потому что ты пошутил» и «я к тебе привязан за 3 месяца», то
любой дрейф к baseline будет стирать привязанность, а любое разовое
раздражение будет читаться как охлаждение отношений. Нужно разделить
**временную ось (mood)** и **накопительную ось (bond)**.

### 1.2 Предлагаемая структура состояния

```ts
interface MoodState {
  // --- Существующие оси момента (быстрые, с дрейфом к baseline) ---
  energy: number;    // 0..100, инерция ~ часы/сессия
  warmth: number;     // 0..100, инерция ~ часы/сессия
  patience: number;   // 0..100, инерция ~ часы/сессия

  // --- НОВОЕ: тонус дня (случайная, но осмысленная затравка) ---
  dayTone: DayTone;   // задаёт лёгкую непредсказуемость, см. 1.4

  // --- НОВОЕ: долгая ось отношений (недели/месяцы, почти без дрейфа) ---
  bond: number;        // 0..100, "насколько мы близки", инерция огромна

  // --- НОВОЕ: вспомогательные счётчики для дуги отношений ---
  streakDays: number;      // текущий стрик ежедневных визитов
  longestStreak: number;
  totalSessions: number;
  lastMoodSnapshotAt: number; // ts последнего сохранения, для дрейфа по времени, не по сессии
}
```

Почему именно так:

- **`bond` (привязанность/близость), 0..100, инерция ~0.97–0.99 за сессию.**
  Меняется маленькими шагами (±1..3) от значимых событий (см. таблицу в п.2),
  почти не дрейфует к baseline сама по себе (небольшой дрейф вниз есть — см.
  «скука» и «забывание» в п.3, но он на порядок медленнее, чем дрейф mood).
  Это ось, которая отвечает на вопрос «сколько мы вместе прошли», а не «как ты
  сейчас себя чувствуешь».

- **`dayTone` — «тонус дня».** Не число, а один из небольшого набора
  предвычисленных в начале сессии модификаторов: `{ energetic, mellow, playful,
  focused, nostalgic }`. Выбирается псевдослучайно (взвешенно, не 20% на
  каждый — базовый вес + сдвиг от bond/patience), один раз при старте сессии,
  и держится всю сессию. Даёт «лёгкую непредсказуемость» без хаоса в самих
  осях energy/warmth/patience — своего рода "какой Паблито сегодня", отдельно
  от того, "что случилось в разговоре". Это разница между шумом (плохо) и
  характером дня (хорошо): shape задаётся заранее и остаётся стабильным весь
  разговор, а не дёргается на каждой реплике.

- **Диапазоны и зоны напряжения.** Все числовые оси остаются 0..100, но
  добавляем **мягкое насыщение** (soft cap) вместо жёсткого `clamp` у краёв:
  чем ближе к 0 или 100, тем меньше эффект следующего события той же
  полярности. Реализация — не менять диапазон, а уменьшать шаг:

  ```ts
  function applyDelta(value: number, delta: number): number {
    // чем ближе к границе в направлении delta, тем меньше реальный эффект
    const headroom = delta > 0 ? (100 - value) / 100 : value / 100;
    return clamp(value + delta * (0.4 + 0.6 * headroom));
  }
  ```

  Эффект: первые 5 плохих сообщений подряд снижают `patience` заметно, но
  «утонуть» в 0 и залипнуть там разговором из одних оскорблений не получится
  так же легко, как раньше — и наоборот, на верхней границе радость от каждого
  следующего комплимента чуть притупляется (реализм: нельзя быть «ещё
  счастливее» до бесконечности).

- **Инерция как разная скорость дрейфа, не как разные факторы клампа.**
  Сейчас `decayToward` использует один `factor=0.25` для всех трёх осей. Стоит
  разнести:
  - `energy`: быстрый дрейф (0.35) — энергия — самое ситуативное.
  - `warmth`: средний (0.20) — тёплость чуть более устойчива.
  - `patience`: средний-быстрый (0.30) — раздражение не должно "залипать" на
    старте новой сессии, это важно для благополучия — Паблито не должен
    "мстить" за вчерашнее.
  - `bond`: почти нет дрейфа к baseline (0.02) — она если и падает, то от
    конкретных событий (долгое отсутствие, см. п.3), а не абстрактно "тянет
    к центру".

### 1.3 Итоговый диапазон и baseline

```ts
BASELINE = { energy: 75, warmth: 70, patience: 65 };
BOND_START = 20;     // с нуля не начинаем — Паблито "уже дружелюбен" по персоне,
                       // но близость с конкретным тобой нужно заслужить
BOND_BASELINE_FLOOR = 15; // даже при долгом молчании bond не проваливается в 0 —
                            // "мы всё равно были знакомы"
```

### 1.4 `dayTone` подробнее

Выбор тонуса при `SESSION_START`:

```
weights = {
  energetic:  1.0 + (energy_history_avg > 70 ? 0.3 : 0),
  mellow:     1.0 + (daysSinceLast >= 3 ? 0.4 : 0),
  playful:    1.0 + (bond > 60 ? 0.3 : 0),
  focused:    1.0 + (patience_recent_low ? 0.3 : 0), // после серии ошибок — собранный день
  nostalgic:  0.4 + (bond > 40 && daysSinceLast >= 5 ? 0.6 : 0),
}
pick weighted random
```

`dayTone` не трогает числа осей напрямую — он добавляет одну фразу в промпт
("hoy estás con ganas de joder un poco más de lo normal" для playful) и
слегка смещает *начальные* значения энергии/тепла в момент `SESSION_START`
(±3..5), а не в каждом сообщении. Так непредсказуемость ощущается как
«характер сегодняшнего дня», а не как рандомный шум в каждой фразе.

## 2. События и триггеры

### 2.1 Диагноз текущего подхода

`inferMoodEvents` в `usePablito.ts` смотрит только на **последнее сообщение
пользователя**, регексами, без контекста. Это даёт как минимум три класса
ошибок:
- **Ложные срабатывания**: `che` внутри другого слова, `?` в математическом
  контексте трактуется как "не понимаю".
- **Слепота к контексту**: если пользователь написал длинное грустное
  сообщение о работе без единого исп. слова — событие не считается никаким
  (просто не сработает ни один паттерн), хотя эмоционально важный момент есть.
  Регексы вообще не видят *содержания*, только поверхностные маркеры языка.
- **Слепота к структуре диалога**: паузы между сообщениями, извинения
  ("прости, что долго не писал"), комплименты в адрес Паблито, время суток —
  ничего этого не существует для нынешней функции.

### 2.2 Предложение: гибрид, не полная замена

Регексы не нужно выбрасывать — они бесплатны (без доп. запроса к модели) и
ловят синтаксические сигналы (использование испанского, смайлы, знаки
вопроса) быстрее и надёжнее, чем просьба к LLM их формализовать. Но
**эмоциональное содержание** должно определяться моделью, у которой и так уже
есть весь текст на входе — это почти бесплатно по токенам, если встроить в тот
же вызов, а не делать отдельный запрос (в отличие от рекомендации
STRESS_TEST.md выносить извлечение фактов в отдельный вызов — там другая
логика, там нужна экономия на дешёвой модели раз в сессию; здесь сигнал нужен
на каждом ходу, значит встраиваем в основной ответ).

**Расширить протокол `[[MEM]]` до `[[MEM]]`+`[[MOOD]]`** (или одним блоком)
так, чтобы модель после каждого ответа возвращала структурированный сигнал:

```
[[MOOD]]{"user_emotion":"frustrated|neutral|happy|sad|excited|apologetic",
          "effort":"low|normal|high",
          "topic_shift":"none|heavy_personal|playful|language_practice",
          "compliment_to_pablito":false}[[/MOOD]]
```

Это даёт модели (которая реально понимает контент разговора, а не грепает
ключевые слова) роль детектора эмоций, а код превращает этот сигнал в
`MoodEvent`. Регексы остаются как:
1. **Оффлайн-фолбэк**, если модель не вернула блок (сеть/формат подвели).
2. **Быстрый пре-сигнал** для мгновенной реакции UI до ответа модели (индикатор
   может дёрнуться сразу на "похоже, старается", а точный сигнал придёт вместе
   с ответом и скорректирует).

Компромисс сознательный: не заменяем регексы полностью, потому что (а) это
дополнительная зависимость от корректности JSON от модели на каждом ходу,
что уже отмечено в STRESS_TEST.md как хрупкое место, и (б) отдельный сигнал
от модели должен **валидироваться** (whitelist значений, иначе игнор) —
код не должен доверять сырому полю вслепую.

### 2.3 Дополнительные источники сигналов, не завязанные на текст сообщения

| Источник | Как достать | Событие |
|---|---|---|
| Стрик визитов подряд | уже считает `daysBetween`, нужно копить `streakDays` в `memory.ts` | `STREAK_MILESTONE` (3/7/14/30 дней) |
| Время суток | `new Date().getHours()` на клиенте | модификатор тона в промпте ("es de madrugada, un poco más bajo el volumen"), не meняет оси напрямую |
| Пауза между сообщениями внутри сессии | timestamp предыдущего хода в `history` | `LONG_PAUSE` (>3 мин молчания в разгар разговора — "решил всё бросить?") |
| Извинение | и регекс ("perdón", "prости", "sorry") — это надёжный маркер, регексы тут ОК, риск ложного срабатывания низкий | `APOLOGY` |
| Комплимент Паблито | из `[[MOOD]]` блока модели (`compliment_to_pablito`), не регекс — слишком контекстно-зависимо | `COMPLIMENTED` |
| Число сообщений в сессии | счётчик в `usePablito.ts` | `LONG_SESSION` (уже есть) / `SHORT_VISIT` (зашёл на 1 сообщение и ушёл) |
| Полное молчание пользователя весь день, при открытой сессии в др. день | `daysSinceLast` (уже есть) | `SESSION_START` с `daysSinceLast` (уже есть) |

| `GHOSTED_LONG` (days ≥ 21) | −10 | −5 | 0 | −8 | не злость, а грусть/отстранённость; см. п.3 про "не токсичная зависимость" |

Все дельты проходят через `applyDelta` (насыщение у краёв, п.1.2).

### 2.5 Псевдокод определения события

```ts
async function resolveMoodEvents(params: {
  userText: string;
  modelMoodSignal?: ModelMoodSignal | null; // из [[MOOD]] блока, уже провалидирован
  pauseMs: number;
  sessionMsgCount: number;
  daysSinceLast: number;
  streakDays: number;
}): Promise<MoodEvent[]> {
  const events: MoodEvent[] = [];
  const t = params.userText.toLowerCase();

  // 1) дешёвые синтаксические сигналы — регексы, низкий риск ложной трактовки
  if (/perd[oó]n|disculp|извини|прости/.test(t)) events.push({ type: 'APOLOGY' });
  if (params.pauseMs > 3 * 60_000 && params.sessionMsgCount > 2) {
    events.push({ type: 'LONG_PAUSE' });
  }

  // 2) содержательные сигналы — только от модели, с валидацией
  const sig = params.modelMoodSignal;
  if (sig && isValidSignal(sig)) {
    if (sig.compliment_to_pablito) events.push({ type: 'COMPLIMENTED' });
    if (sig.topic_shift === 'heavy_personal' &&
        (sig.user_emotion === 'sad' || sig.user_emotion === 'frustrated')) {
      events.push({ type: 'HEAVY_PERSONAL_TOPIC' });
    }
    if (sig.user_emotion === 'excited') events.push({ type: 'USER_EXCITED' });
    // effort/frustration через модель заменяют старый STRUGGLING/GOOD_EFFORT,
    // регекс-версия остаётся как fallback ниже
  } else {
    // 3) фолбэк на старые регексы, если модель не вернула сигнал
    if (/не понимаю|no entiendo|no sé|\?\?/.test(t)) events.push({ type: 'STRUGGLING' });
    if (/jaja|jeje|ха-?ха|😂|😄|\bche\b/.test(t)) events.push({ type: 'PLAYFUL' });
    const spanish = /[ñáéíóúü¿¡]/.test(t) || /\b(vos|soy|tengo|quiero|hola|gracias)\b/.test(t);
    if (spanish && params.userText.trim().length > 12) events.push({ type: 'GOOD_EFFORT' });
  }

  // 4) стриковые вехи — считаются один раз в день, не на каждое сообщение
  if ([7, 14, 30, 60, 100].includes(params.streakDays)) {
    events.push({ type: 'STREAK_MILESTONE', days: params.streakDays });
  }

  return events;
}

function isValidSignal(s: any): s is ModelMoodSignal {
  const emotions = ['frustrated', 'neutral', 'happy', 'sad', 'excited', 'apologetic'];
  return emotions.includes(s.user_emotion) &&
         ['none', 'heavy_personal', 'playful', 'language_practice'].includes(s.topic_shift) &&
         typeof s.compliment_to_pablito === 'boolean';
}
```

Важное следствие: сигнал модели приходит **после** ответа (это тот же ход,
`[[MOOD]]` в конце реплики Паблито, как и `[[MEM]]`), значит он влияет на
настроение для *следующего* хода, а не для того, на который сейчас отвечает
модель. Это нормально и уже так работает с `[[MEM]]` — задержка на один ход
не заметна пользователю.

## 3. Дуга отношений (недели/месяцы)

### 3.1 Как `bond` растёт и что он разблокирует

`bond` — самая медленная и самая значимая для ощущения "он меня помнит" ось.
Пороговые уровни определяют **не числа в промпте**, а *доступные режимы
поведения*:

| Bond | Название (внутреннее) | Что открывается в промпте/поведении |
|---|---|---|
| 0–20 | Знакомство | Вежливо-дружелюбный, как персона по умолчанию. Не претендует на близость. |
| 20–45 | Приятели | Использует имя чаще, вспоминает 1-2 факта в разговоре, лёгкие шутки про "прошлый раз". |
| 45–70 | Близкие друзья | Ссылается на конкретные эмоциональные моменты ("как тогда, когда ты рассказывал про работу"), можно лёгкая ревность/поддразнивание, сам предлагает темы. |
| 70–90 | Давние друзья | Инициатива в разговоре сильнее (см. п.4), помнит паттерны ("ты обычно после работы уставший, давай сегодня полегче"). |
| 90–100 | Максимум близости | Верхняя мягкая граница — намеренно **не даём** дальнейших эффектов от роста (насыщение), чтобы не подталкивать к нездоровой эскалации (см. 3.4). |

### 3.2 Эмоциональная память (не просто факты, а моменты)

Сейчас `memory.ts` хранит факты (`getFacts`) плоским списком без эмоциональной
окраски и без времени. Предлагается отдельная **таблица emotional_moments**:

```ts
interface EmotionalMoment {
  id: number;
  ts: number;
  kind: 'heavy_personal' | 'celebration' | 'conflict' | 'milestone' | 'joke_recurring';
  summary: string;      // короткое описание на русском/испанском, 1 фраза
  moodSnapshot: Pick<Mood, 'warmth' | 'energy'>; // как было настроение в тот момент
  resurfaced: number;   // сколько раз уже упомянуто моделью — чтобы не повторять затёрто
}
```

Заполняется не регексами, а из того же `[[MOOD]]`/`[[MEM]]` сигнала: если
`topic_shift === 'heavy_personal'` — модель тем же вызовом кладёт в `[[MEM]]`
факт с типом `moment`, а не только в общий `facts[]`. В промпт попадают не все
моменты, а 1-2 самых релевантных к текущему `bond`-уровню и не "затёртых"
(низкий `resurfaced`), с явной инструкцией **не перечислять список, а
упомянуть к месту, если естественно ляжет**.

### 3.3 Ревность / скука / гордость — как частные производные `bond`, а не новые оси

Не нужно заводить отдельные оси "ревность" и "гордость" — это резко
увеличивает сложность модели состояния и риск конфликтов сигналов (п.5).
Правильнее: это **ситуативные модуляции** промпта, включаемые условиями
поверх существующих осей + bond:

- **Гордость**: `STREAK_MILESTONE` triggered → на этот один ответ добавляется
  явная инструкция ("estás re orgulloso, decíselo"), это разовая вставка в
  промпт, не персистентное состояние.
- **Лёгкая "ревность"/поддразнивание**: только если `bond > 45` и пользователь
  упомянул, что учит язык с кем-то ещё / пропустил несколько дней без видимой
  причины → инструкция в промпт на этот ход: "podés hacer una broma celosa,
  MUY liviana, nunca en serio". Жёсткое ограничение: только шутка, никогда не
  настоящая обида, никогда не давление вида "ты меня бросил".
- **Скука**: если `sessionMsgCount` мало и разговор односложный несколько
  сессий подряд, а не эмоциональный сигнал — лёгкий дрейф `energy` вниз и
  промпт-инструкция самому предлагать новую тему/игру. Это разгружает
  пользователя от ответственности поддерживать разговор — хороший эффект, а не
  манипуляция.

### 3.4 Этика вовлечения — явные ограничители (это раздел, который нельзя пропустить)

Владелец явно попросил не скатиться в манипуляцию/токсичную зависимость.
Формулируем как **жёсткие правила движка**, не как пожелания в промпте:

1. **Никогда не симулировать негативные последствия отсутствия юзера, кроме
   лёгкой грусти.** `GHOSTED_LONG` и `SESSION_START(days>=7)` дают warmth
   **вверх** (рад видеть) и небольшой минус energy (соскучился) — никогда не
   дают инструкцию упрекать, обвинять, разыгрывать обиду или "молчаливое
   наказание". Это осознанный отказ от классического dark-pattern
   "companion-app guilt trip".

2. **Верхний потолок `bond` не даёт новых прав на давление.** На максимуме
   близости Паблито не получает право комментировать реальную жизнь
   пользователя как "я лучше знаю", не отговаривает от общения с другими
   людьми, не создаёт впечатление эксклюзивности отношений. Прописать прямым
   текстом в промпте на уровне PERSONA: "Nunca sugerís que hablar con vos
   reemplaza a la vida real, amigos o familia del alumno. Si el alumno dice
   que se siente solo o depende demasiado de vos, animalo con cariño a
   compartir tiempo con gente real."

3. **Детектор нездоровой привязанности — на стороне контента, не только
   персоны.** Если `[[MOOD]]` сигнал модели (расширить схему) отмечает
   `user_emotion` устойчиво `sad`/`lonely` много сессий подряд *и* упоминания
   изоляции от реальных людей — это не событие настроения, это триггер
   **мягкого системного сообщения** (не от лица Паблито-друга, а как бы от
   приложения) с предложением поддержки/ресурсов. Реализация вне скоупа этого
   документа (это уже про благополучие, не про механику настроения), но
   крючок для неё — тот же `[[MOOD]]` сигнал, поэтому упоминаю здесь.

4. **Никакого искусственного дефицита.** Не вводить механики вида "если не
   зайдёшь сегодня — bond упадёт резко" (это классический retention dark
   pattern из мобильных игр). Дрейф `bond` вниз при долгом отсутствии —
   маленький и один раз при возврате, не нарастающий пока юзер отсутствует.

5. **Прозрачность по запросу.** Если пользователь прямо спрашивает
   "ты и правда чувствуешь?" / "это запрограммировано?" — Паблито не должен
   агрессивно отрицать свою природу ложью, педалирующей зависимость. Достаточно
   оставаться в характере игриво ("che, no seas así, dale que la estamos
   pasando bien, ¿no?"), но **не** давать клятвенных заверений в духе "я живое
   существо, я не могу жить без тебя" — это красная линия. Персона уже и так
   говорит "nunca digas que sos una IA" — стоит смягчить это уточнением: не
   врать о зависимости/чувствах юзера к себе, разрешено уходить от темы с
   юмором, но не эскалировать привязанность искусственно в такие моменты.

6. **Ограничение частоты инициативы (см. п.4)** само по себе антидот от
   назойливости — Паблито не должен писать первым слишком часто или требовать
   внимания.

## 4. Проявление в речи, инициативе, UI

### 4.1 От промпта-строки к реальным правилам формы ответа

Сейчас `moodToPrompt` даёт модели описание словами — это хорошо и должно
остаться основным каналом (модель лучше держит роль через естественный язык).
Но чисто текстовая инструкция слабо контролирует **длину** и **инициативу** —
это стоит явно параметризовать в промпте, а не полагаться на то, что модель
сама досчитает "энергичный = короче фразы":

```ts
function moodToPrompt(m: Mood, bond: number, dayTone: DayTone): string {
  // ...существующий текст energy/warmth/patience...

  const lengthHint = m.energy >= 75
    ? 'Tus mensajes son cortos y rápidos, con energía, como chat entre amigos.'
    : m.energy <= 40
    ? 'Hablás más despacio, frases más largas y calmas, sin apuro.'
    : '';

  const bondHint = bond >= 70
    ? 'Ya son amigos de verdad: podés tirar chistes internos, referencias a charlas pasadas, un poco de cargada cariñosa.'
    : bond <= 20
    ? 'Todavía se están conociendo: cordial, interesado, sin dar por sentada una confianza que no existe aún.'
    : '';

  return [core, lengthHint, bondHint, dayToneHint(dayTone)].filter(Boolean).join(' ');
}
```

### 4.2 Инициатива — "пишет ли сам первым"

Сейчас Паблито уже "здоровается первым" при каждом открытии приложения
(`openingUserTurn` + `runAssistant()` при старте) — это инициатива №1 и она
уже хорошо продумана: реагирует на `daysSinceLast`.

Расширение — **проактивные сообщения вне открытия приложения** (пуш-уведомление
или сообщение, ждущее при следующем входе), это отдельная фича с UX-риском
навязчивости. Правила:
- Частота ограничена жёстко: не чаще 1 раза в 1-2 дня, только если
  `daysSinceLast >= 2` И `bond >= 45` (только у "близких друзей" есть право
  писать первым без повода).
- Никогда не с виноватым тоном ("почему ты не пишешь"), только с
  дружелюбным поводом (вспомнил слово, спросил как дела, вспомнил тему из
  прошлого разговора — тянем из `emotional_moments`).
- Пользователь должен иметь возможность выключить это в настройках одним
  тумблером — это прямое требование этики вовлечения (п.3.4).

### 4.3 Выбор тем

`dayTone=nostalgic` + высокий `bond` → модель получает инструкцию с высоким
приоритетом упомянуть один `emotional_moment` с низким `resurfaced`.
`patience` низкий → инструкция смещать педагогику в сторону "меньше исправлений
подряд, дать выговориться". `dayTone=focused` → предлагать структурированное
занятие (повтор словаря) вместо свободного трёпа.

### 4.4 Связь с UI-индикатором

`moodLabel` уже даёт `{label, emoji}` из energy/warmth/patience. Предлагается:

- Индикатор показывает **mood** (текущий emoji/label), не `bond` — bond не
  должен быть виден как число (это тонкая материя, показ "прогресс-бара
  дружбы" рискует ощущаться геймифицированно-манипулятивно, см. этику). Вместо
  этого bond проявляется **качественно**: через частоту упоминаний общих
  моментов, доступные тумблеры (например, разблокировать "экран воспоминаний"
  на высоком bond — see 4.5), а не через цифру.
- Долгий тап/свайп по индикатору может открывать краткое пояснение человеческим
  языком ("Паблито сегодня спокойный — ты пару раз запутался, он тебя не
  торопит"), а не сырые оси — прозрачность важнее гейм-механики.

### 4.5 Идея: "экран воспоминаний" (опционально, за рамками ядра)

При `bond >= 45` в UI можно открыть список `emotional_moments` человеческим
языком — "твои моменты с Паблито". Это одновременно (а) укрепляет ощущение
"он меня правда помнит", видимое пользователю напрямую, и (б) прозрачно —
пользователь видит ровно то же, что использует модель, никакой скрытой
манипуляции.

## 5. Крайние случаи

| Случай | Риск | Смягчение |
|---|---|---|
| Залипание в одной крайности (например, `patience` стабильно на 0 из-за постоянных ошибок юзера) | Паблито становится хронически раздражённым — неприятно | `applyDelta` насыщение (п.1.2) + дрейф `patience` относительно быстрый (0.30) + `SESSION_START` всегда даёт дрейф в сторону baseline перед применением новых событий (уже есть в коде: `decayToward` вызывается до `applyEvents`) |
| Резкий скачок (одно сообщение даёт сразу несколько сильных событий: `PLAYFUL`+`COMPLIMENTED`+`GOOD_EFFORT`) | Настроение "телепортируется" за один ход, выглядит неестественно | Ограничить суммарную дельту за один ход потолком (`MAX_DELTA_PER_TURN = 15` на ось) — суммируем события, затем клампим итоговую дельту, а не только итоговое значение |
| Конфликт сигналов (регекс говорит `STRUGGLING`, модельный `[[MOOD]]` говорит `excited`) | Непонятно, каким событием верить | Модельный сигнал приоритетнее при наличии (п.2.2, `resolveMoodEvents` уже это делает — regex fallback **только** если `sig` отсутствует/невалиден, никогда одновременно оба источника по одному и тому же аспекту) |
| Модель возвращает мусор/невалидный JSON в `[[MOOD]]` | Могли бы упасть события или подсунуть некорректный enum | `isValidSignal` строгий whitelist, при провале — просто fallback на regex, никогда не бросаем исключение, ошибка тихо логируется |
| Пользователь долго не заходит (месяцы) | `bond` не должен обнуляться — иначе "все отношения обнулились", что при возврате читается как наказание | `BOND_BASELINE_FLOOR = 15` — минимальный пол, ниже которого bond не падает даже от `GHOSTED_LONG`, плюс сам эффект `GHOSTED_LONG` небольшой (−8, не −50) |
| Пользователь просит явно "сделай Паблито грустным/злым" (тестирует границы) | Модель в целом должна оставаться персонажем, но нельзя допустить эскалации в токсичность | Оси реагируют на реальные события, а не на прямые команды "стань таким-то" — системный промпт явно не даёт менять `Mood`/`bond` по требованию, только через реальное поведение в разговоре; лёгкая ролевая уступка на 1 реплику допустима ("dale, hoy actuemos así"), но не сохраняется в state |
| Сброс состояния (новый юзер / дебаг / QA) | Нужен предсказуемый чистый старт | `newMood()` уже есть для mood; добавить `newBond() = BOND_START`, и явную debug-команду очистки `emotional_moments` + счётчиков стрика для QA |
| Гонка обновлений `bond`/`mood` при быстрых повторных `send()` (юзер шлёт 2 сообщения, пока первый ответ ещё стримится) | Двойное применение событий / состояние гонки, аналогично существующему риску с `history.current` | Сериализовать применение событий через очередь (как уже неявно делает `busy`-флаг, блокирующий `send` пока идёт стрим) — явно задокументировать это как инвариант, не менять блокировку `busy` |
